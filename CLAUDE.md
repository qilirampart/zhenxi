## 五、开发规则

### 5.1 写代码之前

- 必须先确认当前阶段和对应任务
- 不能在未读设计文档相关章节的情况下实现模块
- 模块接口必须对齐 `douyin_prompt_extractor_design.md` 中的数据模型定义

### 5.2 写代码时

- 每个模块独立文件，职责单一
- 不允许在 UI 层写业务/算法逻辑
- OCR 调用必须经过 `app/core/ocr/engine.py` 封装，不得直接调用
- 所有参数配置集中在 `app/config/settings.py`，不散落在各模块
- 文本合并/去重逻辑必须在 `app/core/text/` 中，不得写在 extractor 里

### 5.3 写代码后

- 每完成一个函数/模块，更新 `.codex/progress.md` 日志区
- 遇到错误解决后，写入 `memory/errors.md`
- 发现有效模式，写入 `memory/patterns.md`
- 如需安装依赖，必须先考虑用户可能正开启 Clash/代理；安装失败时优先尝试镜像源，其次尝试临时禁用代理后重试，并记录最终有效方案

### 5.4 禁止行为

- 禁止在一次会话里跨越两个以上 Build 阶段
- 禁止删除 `memory/` 和 `.codex/progress.md` 的历史记录
- 禁止在不对齐设计文档的情况下修改数据模型
- 禁止将 PaddleOCR 调用散落到多处，必须统一封装
- 禁止调用任何 `ccg:*` 技能（包括但不限于 `ccg:init`、`ccg:plan`、`ccg:execute` 等），本项目后续统一使用原生工具与本地规则文件推进，避免因未完善能力导致卡死或中断

## 六、质量门槛

每个 Build 阶段完成前，必须通过以下检查：

| 检查项 | 要求 |
|--------|------|
| 模块接口 | 与设计文档中数据模型一致 |
| 功能可用 | 主流程可跑通，不崩溃 |
| 错误提示 | 异常场景有明确提示，不暴露原始报错 |
| 进度记录 | `.codex/progress.md` 已更新 |
| 代码位置 | 无跨层逻辑，UI/Service/Core 职责清晰 |

**Build-5 打包验证额外要求：**

- exe 可在无 Python 环境的 Windows 机器上运行
- OCR 模型离线可用
- 常见比例视频（9:16 / 16:9 / 1:1）可正常处理

## 七、防中断机制

### 7.1 会话中断前

Claude 在会话结束前必须完成：

1. 将当前完成/未完成任务写入 `.codex/progress.md` 日志区
2. 确认下一步任务明确写入看板区
3. 若有新的错误或模式，写入 `memory/`

### 7.2 会话中断后恢复

执行「二、新会话恢复流程」，然后向用户确认：

```
上次完成到：[具体模块/函数]
当前状态：[Build-X]
下一步：[具体任务]
是否继续？
```

## 八、上下文控制

- 单次会话内，代码变更控制在一个 Build 阶段范围内
- 如果任务超大，拆分为多个子任务，每次只做一个
- 读取文件时优先读关键章节，避免一次读取过多无关内容
- 不要在一次对话中同时推进设计讨论和写代码

---

## 九、经验固化机制

每次 Build 阶段结束后，执行固化流程：

```
1. 回顾本阶段遇到的问题
2. 有效解法写入 memory/errors.md 或 memory/patterns.md
3. 更新 memory/MEMORY.md 摘要区
4. 在 .codex/progress.md 追加阶段完成日志
```

**不得覆盖已有经验记录，只能追加。**
