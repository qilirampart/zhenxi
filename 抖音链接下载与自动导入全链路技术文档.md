# 抖音链接下载与自动导入全链路技术文档

## 1. 文档目标

本文档说明当前项目中新增的“抖音分享链接 -> 视频下载 -> 自动导入 -> 提词提取”全链路实现，重点覆盖：

- 业务目标与链路边界
- 当前实现的技术流程
- 抖音平台“开放下载 / 未开放下载”的差异分析
- 我们为什么仍然能下载部分“未开放下载”的作品
- 关键模块职责
- 配置、异常、风险与后续扩展方向

本文档面向后续开发者，目的是让新成员可以直接基于现有实现继续扩展，而不是重复猜测链路。

---

## 2. 当前业务目标

原始版本的提取工具要求用户先手动下载或准备本地视频，再点击“导入视频”进入提取流程。这样做功能可用，但自动化程度不够。

本轮新增目标是：

1. 用户只需要复制抖音分享文案或分享链接。
2. 工具自动从剪贴板中识别 `https://v.douyin.com/...` 短链。
3. 工具解析可播放视频地址并下载到本地。
4. 下载完成后自动导入当前工作台。
5. 用户可以直接继续候选帧生成、OCR 提取和结果校对。

当前链路不包含以下内容：

- 登录抖音账号
- 调用抖音 App 内部官方“保存本地”按钮
- 批量下载多个链接
- 自动监听剪贴板并静默下载
- 版权合规判断

---

## 3. 全链路总览

### 3.1 业务流

```text
用户复制抖音分享文案
    ->
工作台读取剪贴板文本
    ->
从文本中提取 v.douyin.com 短链
    ->
调用解析服务展开短链并获取结构化数据
    ->
从解析结果中挑选最高置信度的视频播放地址
    ->
请求视频播放地址并保存为本地 mp4
    ->
自动导入 VideoLoader
    ->
视频预览 / 候选帧生成 / OCR 提取
    ->
结果展示与截图保存
```

### 3.2 技术流

```text
分享文本
    ->
DouyinDownloadService.extract_share_url()
    ->
DouyinDownloadService._resolve_share_url()
    ->
解析服务返回 JSON 或直链
    ->
DouyinDownloadService._extract_video_url()
    ->
DouyinDownloadService._download_file()
    ->
output/downloads/*.mp4
    ->
MainWindow._import_video_file()
    ->
VideoLoader.open()
    ->
ExtractionService / StaticExtractor / OCREngine
```

## 5. 下载链路详细设计

## 5.1 分享文本识别

用户复制的通常不是纯链接，而是类似下面的整段分享文案：

```text
6.41 复制打开抖音，看看【DEAIGC的作品】Mrs Pan & Mr Seaman 临终诀别·... https://v.douyin.com/sucYBiXxaVw/ 09/09 qEU:/ F@H.iC
```

当前实现使用正则：

```python
_URL_PATTERN = re.compile(r"https?://[^\\s]+", re.IGNORECASE)
```

在整段文本中抓取第一个 `http/https` URL，并去掉尾部可能附带的 `.,);` 等标点。

入口函数：

```python
DouyinDownloadService.extract_share_url(text)
```

输出示例：

```text
https://v.douyin.com/sucYBiXxaVw/
```

### 设计原因

- 抖音分享文案格式不稳定，但短链通常稳定存在。
- 不依赖固定中文文案模板，兼容性更强。
- 先抽出 URL，再做后续解析，模块职责清晰。

---

## 5.2 短链展开与解析服务调用

入口函数：

```python
DouyinDownloadService._resolve_share_url(share_url)
```

当前默认解析服务配置在：

- [downloader_config.json](E:/点众/自动化工具/runtime/downloader_config.json)

默认值：

```json
{
  "enabled": true,
  "parser_base_url": "https://douyin-vd.vercel.app/api/hello",
  "timeout_seconds": 45,
  "user_agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/123.0.0.0 Safari/537.36"
}
```

请求组装逻辑：

```text
parser_base_url + ?url=<分享短链>&data=
```

示例：

```text
https://douyin-vd.vercel.app/api/hello?url=https%3A%2F%2Fv.douyin.com%2FsucYBiXxaVw%2F&data=
```

### 当前策略

- 不在客户端内直接实现抖音网页协议解析。
- 先依赖可替换的解析服务。
- 解析服务返回 JSON 或直接返回视频 URL。

### 这样设计的原因

1. 抖音短链、跳转页、页面结构经常变化。
2. 解析逻辑独立到服务层，方便以后换源。
3. UI 和解析实现解耦，后续切换到自建服务时无需重写界面。

---

## 5.3 解析结果筛选视频地址

解析服务返回的数据结构未必统一，因此当前实现不写死字段名，而是遍历整个 JSON 树。

核心逻辑：

```python
DouyinDownloadService._walk(payload)
DouyinDownloadService._score_candidate(key_path, url)
DouyinDownloadService._extract_video_url(payload)
```

### 筛选原则

遍历所有字符串值，如果满足：

- 是 `http://` 或 `https://` 开头
- 路径或字段名中更像视频地址

则进入候选集合。

评分示意：

- key path 含 `video`：`+30`
- key path 含 `nwm / no_water / nowater / play`：`+25`
- URL 以 `.mp4` 结尾：`+20`
- URL 本身含 `video`：`+10`
- URL 含 `watermark`：`-10`
- key path 含 `cover / music / avatar / image / images`：直接淘汰

### 目的

解析服务的返回格式可能来自第三方项目，结构不稳定。遍历加评分，比硬编码 `payload["video"]["play_addr"]` 更稳。

---

## 5.4 视频下载与本地保存

入口函数：

```python
DouyinDownloadService._download_file(source_url, target_path)
```

下载行为：

1. 对选中的视频 URL 发起 HTTP 请求。
2. 请求头包含普通浏览器 `User-Agent`。
3. 下载视频时额外带上：

```text
Referer: https://www.douyin.com/
```

4. 以 128 KB 分块流式写入文件。
5. 若下载中断，删除半截文件。

输出路径由下面函数生成：

```python
build_download_output_path(seed_name, suffix)
```

默认目录：

- [output/downloads](E:/点众/自动化工具/output/downloads)

命名格式：

```text
{标题或aweme_id}_{时间戳}.mp4
```

示例：

```text
output/downloads/7622157033089524337_20260328_165229.mp4
```

## 6. 自动导入链路详细设计

## 6.1 UI 入口

主窗口顶部新增按钮：

- `导入本地视频`
- `粘贴链接并下载`

位于：

- [main_window.py](E:/点众/自动化工具/app/ui/main_window.py)

入口函数：

```python
MainWindow._handle_download_from_clipboard()
```

流程：

1. 从剪贴板读取文本。
2. 尝试提取抖音 URL。
3. 如果剪贴板里没有 URL，则弹出多行输入框手动粘贴。
4. 创建后台线程 `_LinkDownloadWorker`。
5. 在线程里执行下载服务，避免阻塞 UI。

---

## 6.2 下载线程

后台线程类：

```python
class _LinkDownloadWorker(QThread)
```

它负责：

- 在线程内调用 `DouyinDownloadService.download_from_text()`
- 把解析和下载进度通过 `progress_text` 信号回传到结果日志面板
- 下载完成后发出 `finished`
- 失败时发出 `failed`

### 为什么要用线程

如果解析和下载直接在主线程里执行：

- 界面会卡死
- 用户会误以为程序失去响应
- 进度日志无法即时更新

---

## 6.3 下载完成后自动导入

下载成功后，主窗口回调：

```python
MainWindow._on_download_finished(result)
```

然后调用：

```python
MainWindow._import_video_file(result.local_path, source_hint=...)
```

这一步与手动导入本地视频共用同一条视频导入逻辑，保证行为一致。

共用逻辑包含：

1. 重置旧状态
2. 打开视频
3. 读取首帧
4. 更新预览
5. 更新视频元信息卡片
6. 初始化时间轴
7. 启用后续提取按钮

### 设计意义

这是当前实现里最关键的一点：

- “链接下载导入”
- “本地文件导入”

并没有两套不同的视频处理逻辑，而是统一走 `_import_video_file()`。

这能减少：

- 状态不一致
- 重复代码
- 一个入口能用、另一个入口不能用的分叉 bug

## 8. 抖音“关闭下载权限”与我们仍可下载的原因

这一段是业务判断的关键，后续开发时不要把“官方未开放下载”误解成“技术上不可访问视频流”。

## 8.1 官方下载能力和播放能力不是一回事

抖音平台上的“允许下载 / 不允许下载”，通常是产品能力层的权限控制，含义更接近：

- 是否在 App UI 中显示“保存本地”
- 是否开放官方下载链路
- 是否允许直接提供面向普通用户的导出体验

但视频只要还能播放，客户端通常仍然需要拿到媒体流。

### 结论

`官方不开放下载` 不等于 `视频字节流在技术上完全不可访问`。

---

## 8.2 我们当前走的是哪条路径

我们当前走的不是：

```text
抖音 App -> 点击保存本地 -> 官方下载接口
```

我们走的是：

```text
分享短链 -> 解析 -> 找到可播放视频地址 -> 直接请求媒体流 -> 本地保存
```

也就是说，我们依赖的是“播放链路可访问”，而不是“官方下载能力开放”。

---

## 8.3 为什么未开放下载的视频有时仍然能保存

常见原因如下：

1. 平台把“播放权限”和“保存权限”拆开控制了。
2. 视频仍然存在可访问的播放 URL。
3. 当前播放 URL 没有被 DRM 或强鉴权完全锁死。
4. 第三方解析服务成功还原了可以直接请求的媒体地址。

### 这类地址通常具备的特征

- 指向 `iesdouyin.com` 等播放接口
- 可被普通 HTTP 请求访问
- 返回视频媒体流
- 请求时只需要基础请求头或简单 Referer

---

## 8.4 为什么这种方式不总是有效

这套方法不是永久稳定，也不是对所有内容都适用。若平台进一步加强保护，链路可能失效。

可能的失效原因：

1. 播放地址强绑定登录态
2. 播放地址强绑定设备指纹
3. URL 改成极短时效签名
4. 视频分片改为加密播放
5. 引入 DRM
6. 服务端加强对 App 内部请求参数校验

### 开发上要保持的认知

当前可用，不代表长期稳定；这是一条工程可行链路，不是平台保证的官方接口。

## 9. 异常处理设计

## 9.1 分享文本里没有链接

抛出：

```python
DouyinDownloadError("没有识别到抖音分享链接...")
```

UI 侧处理：

- 剪贴板无链接时，弹出多行文本框让用户手动粘贴

## 9.2 解析服务不可用

表现：

- 网络失败
- 服务返回非 JSON
- 服务未返回视频地址

处理：

- 抛 `DouyinDownloadError`
- UI 结果日志中追加错误
- 状态徽章切换为失败

## 9.3 视频下载中断

处理：

- 删除半截文件
- 避免本地留下无效 mp4

## 9.4 下载成功但导入失败

处理：

- 保留已下载文件
- 阻止进入提取流程
- 通过 `VideoLoaderError` 提示导入失败

## 11. 当前实现的优点

1. 解析服务与 UI 解耦，后面可直接换源。
2. 下载后自动导入，不再要求用户自己找文件。
3. 本地导入与下载导入共用一套视频载入流程，维护成本低。
4. 下载链路在后台线程里执行，不阻塞桌面界面。
5. 解析结果字段不写死，对第三方返回格式更宽容。

---

## 12. 当前实现的缺点与风险

1. 依赖第三方解析服务，不完全可控。
2. 解析结果字段不稳定时，仍可能选错 URL。
3. 目前没有下载质量切换能力。
4. 没有批量下载。
5. 没有自动监听剪贴板。
6. 没有缓存解析结果。
7. 法律、平台规则、版权合规风险需要额外评估。

## 14. 一页总结

当前项目新增的自动化下载能力，本质上不是“调用抖音官方下载”，而是：

```text
分享文案
    ->
提取短链
    ->
解析服务展开并返回结构化结果
    ->
从结果中挑出视频播放地址
    ->
直接请求媒体流保存到本地
    ->
自动导入现有视频提词流程
```

平台“未开放下载”限制的通常是官方下载能力，不一定等于技术上无法拿到播放流。只要视频仍可播放，且播放流未被更严格的 DRM / 鉴权机制完全保护，就可能仍然被下载并导入本地工作流。

这也是当前系统能够同时下载“开放下载”和“未开放下载”两类作品的根本原因。
