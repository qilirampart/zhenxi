# 抖音视频提示词提取工具详细设计文档

## 1. 文档信息

- 文档类型：详细设计文档
- 主要读者：当前开发者、Claude 后续开发上下文
- 文档目标：指导首版开发，并为后续功能扩展提供稳定架构基础
- 运行平台：Windows
- 运行方式：本地离线运行，优先 CPU
- 处理对象：本地抖音视频文件

---

## 2. 项目背景

短视频中经常包含可复用的提示词内容，主要表现为两类：

1. **静态提示词**：提示词直接展示在视频画面中，通常停留若干秒，可通过单帧或少量帧识别。
2. **滚动提示词**：提示词以滚动或分页方式展示，需要按一定频率连续截图，再将多帧 OCR 结果合并、去重并还原为完整文本。

本项目目标是开发一个 Windows 桌面工具，支持上述两类视频提示词提取，并最终打包为可分发的 `.exe` 供其他人直接使用。

---

## 3. 产品目标与非目标

### 3.1 产品目标

首版产品需要满足以下目标：

1. 支持导入本地视频文件。
2. 支持抖音常见视频比例预览与处理。
3. 支持两种核心模式：
   - 模式 A：静态提示词提取
   - 模式 B：滚动提示词提取
4. 支持 OCR 识别中文提示词。
5. 支持结果整合、去重与导出。
6. 支持打包成离线可运行 exe。
7. 架构上预留后续扩展能力。

### 3.2 非目标

以下内容不纳入首版范围：

1. 直接通过抖音链接下载视频。
2. 批量处理多个视频。
3. 云端 OCR 或联网服务。
4. 音频识别、语音转文字。
5. 自动发布、账号登录、平台交互等非提取能力。
6. macOS / Linux 兼容。

## 4. 需求范围

### 4.1 核心功能范围

#### 模式 A：静态提示词提取

- 支持自动选帧。
- 支持手动拖动进度条预览视频帧。
- 支持用户确认当前帧并加入识别列表。
- 支持自动选帧与手动补充并存。
- 自动模式默认输出 5 张候选帧。
- 对选中的一个或多个帧执行 OCR。
- 对多帧 OCR 结果做去重与合并。

#### 模式 B：滚动提示词提取

- 支持用户选择截图频率：2 秒 / 3 秒。
- 按设定频率对视频自动抽帧。
- 对全部帧执行 OCR。
- 支持多帧文本去重与顺序合并。
- 同时输出：
  - 合并后的完整长文本
  - 按截图顺序分段文本

### 4.2 通用功能范围

- 支持视频比例自适配预览。
- 支持手动框选 OCR 区域。
- 支持保留抽取截图到本地。
- 支持导出 TXT 和 JSON。
- 支持离线运行。
- 支持处理进度展示。
- 支持错误提示与日志记录。

## 5. 用户场景

### 场景 1：提取静态展示的提示词

用户导入一个抖音视频，程序自动给出 5 张候选关键帧。用户检查后，如有遗漏，拖动进度条找到更合适的画面，点击“添加当前帧”，然后执行识别并导出结果。

### 场景 2：提取滚动展示的提示词

用户导入一个滚动提示词视频，选择“每 2 秒截图”或“每 3 秒截图”，程序自动抽帧、识别并整合输出完整提示词，同时保留按帧顺序的分段文本供校对。

### 场景 3：排除无关 UI 干扰

用户发现抖音右侧按钮、底部昵称或顶部标题影响识别，可使用手动框选功能限制 OCR 区域，仅识别提示词主体部分。

## 6. 非功能需求

### 6.1 平台要求

- 仅支持 Windows。
- 打包后无需用户安装 Python 环境。
- 优先兼容无 GPU 的普通办公电脑。

### 6.2 性能要求

- 10 分钟以内的常见短视频应可正常处理。
- CPU 环境下应保持可接受的交互反馈，避免界面冻结。
- 抽帧、OCR、文本整合过程应有进度反馈。

### 6.3 可用性要求

- 界面应简单，主流程清晰。
- 用户无需了解技术细节即可完成提取。
- 出错信息应明确到步骤级别，例如“视频加载失败”“OCR 初始化失败”“导出失败”。

### 6.4 可扩展性要求

架构需允许后续扩展以下能力：

- 新增提取模式
- 新增导出格式
- 新增批量处理
- 新增自动识别区域策略
- 新增更多截图频率配置

## 7. 技术栈选型

## 7.1 最终技术栈

- 语言：Python 3.11
- GUI：PySide6
- 视频读取与图像处理：OpenCV
- 视频解码兼容与辅助抽帧：FFmpeg
- OCR：PaddleOCR（CPU 版）
- 文本相似度与去重：RapidFuzz
- 数据处理：Python 标准库 + dataclasses + pathlib + json
- 打包：PyInstaller

### 7.2 选型原因

#### Python

适合视频处理、OCR 调用、文本合并和快速桌面工具开发，生态完整，后续扩展成本低。

#### PySide6

适合开发 Windows 桌面 GUI，界面能力明显优于 Tkinter，后续扩展手动框选、结果编辑、批量任务列表等功能更方便。

#### OpenCV

适合读取视频元数据、按时间取帧、图像增强、裁剪与清晰度分析。

#### FFmpeg

用于增强复杂编码视频的兼容性，并可作为 OpenCV 取帧失败时的兜底方案。

#### PaddleOCR

中文场景效果较稳，适合字幕、海报字、提示词文本等复杂视频画面识别。

#### RapidFuzz

适合多帧文本之间的相似度比较、重复段落判断和滚动文本合并。

#### PyInstaller

适合首版快速完成 exe 分发。建议使用 onedir 模式，降低模型文件、ffmpeg 和依赖打包复杂度。

## 8. 总体架构设计

系统采用分层设计，降低 GUI、视频处理、OCR 和文本处理之间的耦合。

```text
UI 层
  ├─ 主窗口
  ├─ 视频预览与进度条
  ├─ 模式参数面板
  ├─ 手动框选组件
  └─ 结果展示与导出

应用服务层
  ├─ 任务调度
  ├─ 模式控制器
  ├─ 进度通知
  └─ 导出服务

核心能力层
  ├─ 视频加载与抽帧
  ├─ 图像预处理
  ├─ OCR 封装
  ├─ 静态提示词提取器
  ├─ 滚动提示词提取器
  └─ 文本清洗/合并/去重

基础设施层
  ├─ 文件系统
  ├─ 日志
  ├─ 配置
  ├─ 临时目录管理
  └─ FFmpeg 调用
```

### 8.1 设计原则

1. **界面与算法解耦**：UI 只负责参数收集、进度展示和结果呈现。
2. **模式能力独立**：静态模式与滚动模式分别实现，便于维护和扩展。
3. **OCR 引擎统一封装**：未来可替换为其他 OCR 实现。
4. **结果模型统一**：无论来源于哪种模式，最终统一输出结构化结果。
5. **优先保证可用性**：首版优先实现稳定流程，不追求过度自动化。

## 10.2 视频预览区

视频预览区为主流程核心组件，职责包括：

- 显示当前视频帧
- 支持拖动进度条
- 显示当前时间点
- 支持在模式 A 中手动选帧
- 支持在任一模式下进行手动框选 OCR 区域

### 10.2.1 比例适配规则

支持以下常见比例：

- 9:16
- 16:9
- 1:1
- 3:4
- 4:3

显示时采用“等比缩放 + 居中 + 黑边补齐”策略，不拉伸、不裁切。

### 10.2.2 视频预览实现策略

首版不依赖复杂播放器，采用以下策略：

- 使用 OpenCV 读取视频和指定时间点帧
- 使用自定义 Slider 控制时间轴
- 拖动时按时间点读取目标帧并刷新预览
- 如遇特定编码 seek 不准确，可引入 FFmpeg 辅助取帧

该方案更适合本项目的“选帧和截图”需求，复杂度低于完整播放器实现。

## 10.3 模式 A 面板设计

### 参数项

- 选帧方式：
  - 自动选帧
  - 手动选帧
  - 自动 + 手动补充（默认推荐）
- 候选帧数量：固定 5 张（首版）
- 是否启用手动框选 OCR 区域
- 是否保留截图

### 交互流程

1. 用户导入视频。
2. 程序自动分析并生成 5 张候选关键帧。
3. 用户可直接勾选候选帧。
4. 用户也可拖动进度条，定位到任意时刻点击“添加当前帧”。
5. 所有已选帧进入待识别列表。
6. 点击“开始提取”后执行 OCR。

## 10.4 模式 B 面板设计

### 参数项

- 截图频率：
  - 每 2 秒一张
  - 每 3 秒一张
- 是否启用手动框选 OCR 区域
- 是否保留截图

### 交互流程

1. 用户导入视频。
2. 选择截图频率。
3. 程序根据视频时长估算待处理帧数。
4. 点击“开始提取”。
5. 抽帧、OCR、文本合并依次执行。
6. 结果区显示完整文本与按帧分段文本。

## 10.5 结果区设计

结果区建议分为两个 Tab：

### Tab 1：完整结果

- 展示合并去重后的最终提示词文本
- 支持用户直接编辑
- 支持复制和导出

### Tab 2：按帧结果

- 按时间顺序展示每帧 OCR 结果
- 每项包含：
  - 时间戳
  - 截图缩略图路径
  - 原始 OCR 文本
  - 清洗后文本
- 便于用户校对和追溯

## 11. 模式 A 详细流程设计

## 11.1 输入

- 本地视频文件
- 选帧方式配置
- 可选手动框选区域
- 是否保留截图

## 11.2 自动选帧流程

### 步骤 1：低频采样

- 按 0.5 秒或 1 秒间隔采样视频帧。
- 生成候选帧集合。

### 步骤 2：画面稳定度检测

- 计算相邻采样帧的帧差。
- 若帧差较小，说明画面相对稳定，可作为候选。

### 步骤 3：清晰度评分

- 对候选帧计算 Laplacian 方差，作为清晰度指标。
- 清晰度高的帧优先。

### 步骤 4：文字密度评分

可采用两种实现方式，首版建议使用简化版：

- 简化版：通过边缘密度、连通域面积估计文字区域丰富度
- 增强版：用 OCR detection 或轻量文本检测估计文本占比

首版建议采用“清晰度 + 稳定度 + 文本区域面积”综合评分。

### 步骤 5：候选帧筛选

- 按综合分排序
- 去除时间上过于接近的重复帧
- 输出最多 5 张候选帧

## 11.3 手动选帧流程

1. 用户拖动时间轴。
2. 程序显示当前时间点对应帧。
3. 用户点击“添加当前帧”。
4. 当前帧写入待识别列表。
5. 支持删除已选帧。

## 11.4 OCR 识别流程

1. 对所有待识别帧按 ROI 裁剪。
2. 执行图像预处理。
3. 调用 OCR 引擎识别。
4. 对输出文本进行清洗。
5. 对多帧结果去重合并。

## 11.5 输出

- 完整合并文本
- 按帧 OCR 明细
- 可选截图目录
- TXT/JSON 导出文件

## 12. 模式 B 详细流程设计

## 12.1 输入

- 本地视频文件
- 截图频率（2s / 3s）
- 可选手动框选区域
- 是否保留截图

## 12.2 抽帧流程

### 步骤 1：时间点生成

根据视频时长和截图频率生成时间点序列，例如：

- 频率 2 秒：0s, 2s, 4s, 6s...
- 频率 3 秒：0s, 3s, 6s, 9s...

### 步骤 2：按时间点抽帧

- 使用 OpenCV 定位时间点并取帧
- 如特定视频出现 seek 偏移，可切换 FFmpeg 取帧

### 步骤 3：截图保留

若用户启用“保留截图”，则按统一命名规则保存：

`frame_0001_00-00-02.jpg`

## 12.3 OCR 与文本合并流程

1. 对每张截图裁剪 ROI。
2. 执行图像预处理。
3. 调用 OCR。
4. 清洗无关内容。
5. 对相邻帧文本做相似度比对。
6. 合并新增内容，删除重复内容。
7. 生成：
   - 合并后的完整文本
   - 按帧分段文本

## 12.4 滚动文本合并策略

滚动文本场景的关键在于“新增内容识别”。

### 基本规则

1. 如果后一帧与前一帧高度相似，保留更完整版本。
2. 如果后一帧是前一帧的后缀扩展，则做拼接。
3. 如果后一帧明显进入新段落，则作为新块处理。
4. 对整体结果再次做全局去重。

### 建议算法

#### 规则 1：完全重复去重

使用标准化文本后做完全相等判断。

#### 规则 2：高相似文本去重

使用 RapidFuzz 比较相邻帧文本相似度：

- 相似度 >= 92：视为重复或轻微变化，保留更长文本
- 相似度在 70~92：进一步做重叠分析
- 相似度 < 70：视为新段落或新内容

#### 规则 3：前后缀重叠合并

计算两段文本的最长重叠部分：

- 若 `A` 结尾与 `B` 开头重叠，则拼接为 `A + B[overlap:]`
- 若 `B` 基本包含 `A`，保留 `B`

#### 规则 4：按行去重

对于多行文本：

- 先按行拆分
- 删除重复行
- 保留首次出现顺序

### 输出形式

#### 完整文本

用于最终复制、导出和直接使用。

#### 按帧分段文本

用于用户核对滚动过程，便于发现漏识别和拼接错误。

## 13. OCR 区域与图像预处理设计

## 13.1 OCR 区域策略

首版支持两种区域模式：

### 自动区域

根据视频比例套用默认安全区域，尽量避开抖音 UI 干扰。

#### 9:16 竖屏默认建议

- 忽略顶部约 10%
- 忽略底部约 18%
- 忽略右侧约 15%
- OCR 主区域位于中间主体区域

#### 16:9 横屏默认建议

- 忽略顶部约 8%
- 忽略底部约 12%
- 忽略左右边缘各约 5%

#### 1:1 / 3:4 / 4:3

采用居中大区域策略，具体参数写入配置文件，后续可调。

### 手动框选区域

用户在预览区拖拽矩形框，指定 OCR 区域。

#### 规则

- 坐标基于预览画面
- 需映射回原始视频分辨率坐标
- 手动框选优先级高于自动区域
- 同一视频处理周期内保持同一 ROI

## 13.2 图像预处理

OCR 前可进行以下预处理：

1. 转灰度
2. 适度放大
3. 双边滤波/中值滤波去噪
4. 对比度增强
5. 自适应阈值二值化（按需开启）
6. 倾斜修正（后续扩展）

首版建议提供固定预处理管线，不做复杂配置面板。

## 14. 文本清洗与格式化设计

OCR 结果需要清洗常见干扰文本。

## 14.1 清洗目标

- 抖音水印
- 账号昵称
- 点赞/评论/分享等 UI 文本
- 时间、电量等系统状态字样（如果录屏中存在）

## 14.2 清洗策略

### 规则清洗

- 去掉首尾空白
- 合并多个空格为一个
- 合并连续空行为一个空行
- 去除明显无意义短文本，如单独的“赞”“评论”等

### 黑名单过滤

预留可配置黑名单词表，例如：

- 抖音
- 分享
- 评论
- 点赞
- 原声

### 相似文本归一化

在做去重前，将文本进行标准化：

- 全角转半角
- 统一换行
- 去除尾部标点差异
- 去除多余空白字符

## 19. 打包与分发设计

## 19.1 打包方式

首版采用 PyInstaller `onedir` 模式。

原因：

- 依赖较重，适合目录分发
- OCR 模型与 ffmpeg 文件管理更稳定
- 启动速度优于 onefile

## 19.2 分发目录建议

```text
DouyinPromptExtractor/
├── DouyinPromptExtractor.exe
├── runtime/
│   ├── ffmpeg/
│   │   └── ffmpeg.exe
│   └── models/
│       └── paddleocr/...
├── logs/
└── config.json
```

## 19.3 模型策略

选择“模型随程序一起打包”的离线方案：

- 不依赖首次联网下载
- 更适合发给其他人直接使用
- 体积增大但可接受

## 19.4 FFmpeg 策略

- 将 ffmpeg.exe 随程序一起分发
- 封装统一调用接口
- 仅在 OpenCV 无法稳定取帧时启用

## 22. 风险与应对

## 22.1 主要风险

### 风险 1：OCR 对复杂背景识别不稳定

应对：

- 增加 ROI 限制
- 增加预处理
- 允许用户手动校对结果

### 风险 2：滚动文本合并出现错拼

应对：

- 保留按帧分段结果
- 合并算法分层设计
- 阈值可配置

### 风险 3：OpenCV 对特定视频 seek 不准

应对：

- 增加 FFmpeg 兜底抽帧通道

### 风险 4：CPU 环境速度偏慢

应对：

- 任务后台执行
- 进度条反馈
- 后续可增加缓存和并发优化

### 风险 5：打包体积较大

应对：

- 首版接受较大体积，优先稳定离线可用
- 后续再评估 Nuitka 或模型裁剪

## 24. 首版开发结论

首版方案确定如下：

- 平台：Windows 本地离线
- 技术栈：Python + PySide6 + OpenCV + FFmpeg + PaddleOCR + RapidFuzz + PyInstaller
- 输入：本地视频文件
- 模式 A：自动选帧 + 手动选帧，默认 5 张候选帧
- 模式 B：2 秒 / 3 秒抽帧
- 识别区域：自动区域 + 手动框选
- 输出：完整文本 + 按帧分段文本 + TXT/JSON
- 辅助功能：保留截图、进度显示、日志记录
- 打包方式：离线 onedir exe

该方案能够覆盖当前明确需求，并为后续功能扩展保留足够空间。
